How to Deploy and Publish an API Wrapper for a URL Unshortener Service

In today’s fast-paced digital landscape, shortened URLs are everywhere—from social media posts to marketing campaigns. But behind every "bit.ly" or "tinyurl.com" lies a potential risk: hidden destinations, broken links, or even malicious redirects. A URL unshortener service acts as a transparency layer, revealing the true endpoint before you click. While many web-based unshorteners exist, building and deploying your own API wrapper around such a service gives you control, speed, and integration capabilities. This article walks you through the entire process—from planning your wrapper to publishing it for public or private use—so you can offer a reliable URL unshortener service to users or your own applications.

Why Build an API Wrapper for a URL Unshortener Service?

Before diving into code, it’s worth understanding the "why." An API wrapper abstracts the complexity of a backend service (like a headless browser or a redirect-following engine) into simple, programmatic endpoints. Here’s why you might want one:

  • Security screening – Automatically check links before opening them in a browser.
  • Marketing analytics – Track where shortened campaign URLs actually lead.
  • Content moderation – Verify that user-submitted links don’t point to unsafe sites.
  • Integration ease – Plug unshortening into your existing tools (Slack bots, security scanners, etc.).

By deploying your own wrapper, you avoid rate limits imposed by third-party services, you can customize the output format, and you ensure uptime that aligns with your needs.

Step 1: Choose Your Backend Unshortening Method

Every URL unshortener service relies on following HTTP redirects. The simplest approach uses Python’s requests library with allow_redirects=False to inspect the Location header. However, many URL shorteners (including some on social media) use JavaScript-based redirects or meta-refresh tags. For a robust service, you’ll need a headless browser like Playwright or Selenium.

Recommended stack for a production-ready wrapper:

  • Language: Python 3.9+ (or Node.js if your ecosystem prefers it)
  • HTTP client: httpx (async support) or requests
  • Headless browser: Playwright (lighter than Selenium, supports Chromium/Firefox/WebKit)
  • API framework: FastAPI (async, auto-docs) or Flask (simpler, synchronous)
  • Deployment: Docker + cloud host (Render, Railway, or AWS)

Step 2: Build the Core Unshortening Logic

Let’s create a basic unshortener function in Python. This example uses httpx for simple redirect chains and falls back to Playwright for JavaScript-heavy links.

import httpx
from urllib.parse import urlparse

def unshorten_url(short_url: str, max_redirects: int = 10) -> dict:
    """
    Follow redirects and return final URL, status, and chain.
    """
    result = {
        "original_url": short_url,
        "final_url": None,
        "redirect_chain": [],
        "status": "success"
    }
    try:
        with httpx.Client(follow_redirects=False, timeout=10.0) as client:
            current_url = short_url
            for _ in range(max_redirects):
                response = client.get(current_url)
                result["redirect_chain"].append({
                    "url": current_url,
                    "status_code": response.status_code
                })
                if response.status_code in (301, 302, 303, 307, 308):
                    location = response.headers.get("Location")
                    if not location:
                        break
                    # Handle relative redirects
                    if not urlparse(location).netloc:
                        location = urlparse(current_url)._replace(path=location).geturl()
                    current_url = location
                else:
                    result["final_url"] = current_url
                    break
            else:
                result["status"] = "max_redirects_exceeded"
    except Exception as e:
        result["status"] = f"error: {str(e)}"
    return result

For JavaScript redirects, you’d use Playwright to wait for the page to fully load and then read the final URL:

from playwright.sync_api import sync_playwright

def unshorten_with_browser(short_url: str) -> str:
    with sync_playwright() as p:
        browser = p.chromium.launch(headless=True)
        page = browser.new_page()
        page.goto(short_url, wait_until="networkidle")
        final_url = page.url
        browser.close()
        return final_url

Step 3: Create the API Wrapper

Now wrap the logic in a FastAPI endpoint. This gives you automatic OpenAPI documentation and request validation.

from fastapi import FastAPI, HTTPException, Query
from pydantic import BaseModel

app = FastAPI(title="URL Unshortener Service", version="1.0.0")

class UnshortenResponse(BaseModel):
    original_url: str
    final_url: str | None
    redirect_chain: list[dict]
    status: str

@app.get("/unshorten", response_model=UnshortenResponse)
async def unshorten(
    url: str = Query(..., description="The shortened URL to resolve"),
    use_browser: bool = Query(False, description="Use headless browser for JS redirects")
):
    if not url.startswith(("http://", "https://")):
        raise HTTPException(status_code=400, detail="Invalid URL scheme")
    
    if use_browser:
        final = unshorten_with_browser(url)
        return UnshortenResponse(
            original_url=url,
            final_url=final,
            redirect_chain=[{"url": url, "status_code": 200}],
            status="success"
        )
    else:
        result = unshorten_url(url)
        return UnshortenResponse(**result)

Step 4: Add Practical Features

A generic URL unshortener service becomes truly useful with these enhancements:

  • Rate limiting – Use slowapi to prevent abuse (e.g., 10 requests per minute per IP).
  • Cache layer – Store resolved URLs in Redis or a simple in-memory cache (e.g., cachetools) to avoid re-fetching the same short link.
  • Malware detection – Integrate with Google Safe Browsing API or a local blocklist.
  • Bulk endpoint – Accept a JSON array of URLs and return all results in one request.
  • Logging – Log requests and errors to stdout or a service like Sentry.

Step 5: Containerize and Deploy

Docker simplifies deployment across environments. Here’s a minimal Dockerfile:

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
# Install Playwright browser binaries
RUN playwright install chromium
COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

Deployment options:

  • Render – Free tier with auto-deploy from GitHub; supports Docker.
  • Railway – Simple CLI deploy, handles HTTPS automatically.
  • Fly.io – Global edge deployment with low latency.
  • AWS Lambda + API Gateway – Serverless, but requires packaging Playwright (use playwright-lambda layer).
